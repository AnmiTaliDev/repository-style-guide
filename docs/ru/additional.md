# Дополнительные файлы документации

Опциональные, но очень ценные файлы, которые улучшают качество проекта и опыт пользователей.

## FAQ

Часто задаваемые вопросы — экономит время как для пользователей, так и для мейнтейнеров.

### Требуемое содержание
- Распространённые вопросы от пользователей с чёткими ответами
- Советы по устранению неполадок для известных проблем
- Объяснение проектных решений
- Сравнение с похожими инструментами или альтернативами
- Вопросы, связанные с производительностью
- Вопросы о лицензировании и использовании
- Вопросы о совместимости с платформами
- Известные ограничения и обходные пути

### Когда включать
- После получения повторяющихся вопросов
- Для сложных проектов с кривой обучения
- Проекты с распространёнными подводными камнями или ловушками
- НЕ создавайте пустой FAQ заранее

### Варианты названия
- `FAQ.md`
- `FAQ`

### Организация
- Группируйте по теме или категории
- Сначала наиболее распространённые вопросы
- Чёткий формат вопроса и ответа
- Ссылки на соответствующую документацию для деталей

## EXAMPLES

Шаблоны использования и демонстрации кода.

### Варианты структуры

**Вариант с одним файлом:**
- Все примеры в одном документе
- Организованы по сложности (от базового к продвинутому)
- Чёткие заголовки разделов
- Краткие объяснения с каждым примером

**Вариант с директорией:**
- Директория `examples/` с поддиректориями
- Отдельные файлы для различных случаев использования
- README в каждой поддиректории
- Запускаемые файлы кода
- Примеры данных, если нужны

### Варианты названия
- `EXAMPLES.md` (один файл)
- `EXAMPLES` (один файл)
- `examples/` (директория, обычно в нижнем регистре)

### Требуемое содержание
- Примеры базового использования
- Демонстрации распространённых случаев использования
- Примеры продвинутых функций
- Сценарии реального применения
- Демонстрации обработки ошибок
- Примеры конфигурации

### Лучшие практики
- Делайте примеры запускаемыми как есть
- Включайте комментарии, объясняющие ключевые моменты
- Начинайте с простого, переходите к сложному
- Покрывайте распространённые случаи использования
- Предоставляйте примеры входных/выходных данных
- Держите примеры обновлёнными при изменениях API

## ARCHITECTURE

Высокоуровневая архитектура и документация по дизайну.

### Требуемое содержание
- Обзор системной архитектуры
- Отношения и взаимодействия компонентов
- Используемые паттерны проектирования
- Ключевые абстракции и их назначения
- Поток данных через систему
- Обоснование крупных проектных решений
- Выбор технологий и обоснования
- Соображения масштабируемости
- Архитектура безопасности
- Характеристики производительности

### Когда включать
- Сложные проекты с несколькими компонентами
- Фреймворки и библиотеки
- Проекты, где понимание дизайна важно для контрибьюторов
- Системы с неочевидной архитектурой
- Проекты с конкретными архитектурными паттернами

### Варианты названия
- `ARCHITECTURE.md`
- `ARCHITECTURE`
- `DESIGN.md`
- `DESIGN`

### Цель
- Помогает новым контрибьюторам понять общую картину
- Документирует архитектурные решения
- Облегчает рассуждение о системе
- Предотвращает архитектурный дрейф
- Ресурс для онбординга

## API

Справочник или документация по API.

### Требуемое содержание
- Документация по endpoint'ам (для REST API)
- Документация по функциям/классам (для библиотек)
- Форматы запросов/ответов
- Механизмы аутентификации
- Требования авторизации
- Информация об ограничениях по частоте
- Ответы об ошибках и коды
- Примеры использования для каждого endpoint'а/функции
- Информация о версионировании
- Уведомления об устаревании

### Современные альтернативы
Вместо вручную поддерживаемого API.md:
- Файлы спецификации OpenAPI/Swagger (`openapi.yaml`, `swagger.json`)
- Сгенерированная документация из комментариев кода (JSDoc, Sphinx, Doxygen и т.д.)
- Директория `docs/api/` для обширной документации
- Инструменты интерактивной документации API

### Варианты названия
- `API.md`
- `API`
- `API_REFERENCE.md`
- `openapi.yaml` (для REST API, предпочтительно)
- `swagger.json` (для REST API)
- `docs/api/` (директория)

### Когда использовать API.md
- Простые API с несколькими endpoint'ами
- Быстрая справочная документация
- Не рекомендуется для больших API (используйте сгенерированную документацию)

## MIGRATION

Руководство по обновлению между крупными версиями.

### Требуемое содержание
- Несовместимые изменения, перечисленные чётко
- Пошаговые инструкции по миграции
- Примеры трансформации кода (до/после)
- Предупреждения об устаревании и сроки
- График поддержки старых версий
- Распространённые подводные камни миграции
- Автоматизированные инструменты миграции (если доступны)
- Рекомендации по тестированию после миграции
- Процедуры отката

### Когда включать
- После выпуска v2.0+ с несовместимыми изменениями
- Когда API значительно меняется
- Энтерпрайз ПО с долгоживущими развёртываниями
- Крупный рефакторинг или архитектурные изменения
- Обновления версий зависимостей с несовместимыми изменениями

### Варианты названия
- `MIGRATION.md`
- `MIGRATION`
- `UPGRADING.md`
- `UPGRADING`

### Организация
- Отдельный раздел для каждого перехода между крупными версиями
- Хронологический порядок (сначала новые)
- Чёткие заголовки для каждого несовместимого изменения
- Уровни приоритета (обязательно vs. рекомендуется)

## VERSION

Текущий идентификатор версии проекта.

### Содержание
- Один номер версии
- Опционально: дата релиза
- Опционально: название версии или кодовое имя

### Варианты названия
- `VERSION`
- `VERSION.txt`
- `.version`

### Современные альтернативы (предпочтительно)
Вместо файла VERSION используйте:
- Git теги для версионирования
- Языко-специфичные поля версии:
  - Python: `__version__` в `__init__.py` или `pyproject.toml`
  - Node.js: `version` в `package.json`
  - Rust: `version` в `Cargo.toml`
  - Java: `pom.xml` или `build.gradle`

### Когда использовать файл VERSION
- Когда не существует языко-специфичного файла версии
- Для языко-агностичных проектов
- Когда существует несколько идентификаторов версии
- Не рекомендуется для большинства современных проектов

## FUNDING

Информация о финансировании проекта и спонсорстве.

### Требуемое содержание
- Как спонсировать или пожертвовать
- Доступные платформы спонсорства
- Текущие спонсоры (с разрешения)
- Что поддерживает финансирование (разработка, инфраструктура и т.д.)
- Уровни спонсорства и преимущества
- Информация о налоговой вычитаемости (если применимо)
- Прозрачность об использовании средств

### Интеграция с GitHub
GitHub распознаёт формат `.github/FUNDING.yml`:
- Добавляет кнопку "Sponsor" в репозиторий
- Поддерживает несколько платформ финансирования
- Структурированный формат YAML

### Варианты названия
- `FUNDING.md`
- `FUNDING`
- `FUNDING.yml` (в директории `.github/`, рекомендуется для GitHub)
- `SPONSORS.md`

### Поддерживаемые платформы
- GitHub Sponsors
- Patreon
- Open Collective
- Ko-fi
- Tidelift
- Пользовательские URL

## CODEOWNERS

Автоматически назначает ревьюеров для pull request'ов.

### Цель
- Автоматическое назначение ревьюера
- Определение границ владения кодом
- Обеспечение соответствующего ревью
- Распределение обязанностей по ревью
- Требуемые ревью от назначенных владельцев

### Формат файла
- Строки комментариев начинаются с `#`
- Шаблоны соответствуют файлам или директориям
- Владелец указывается именем пользователя GitHub или командой
- Наиболее конкретный шаблон имеет приоритет
- Поддерживает glob шаблоны

### Требуемое расположение
Должен быть в одном из этих мест:
- Корень репозитория: `CODEOWNERS`
- `.github/CODEOWNERS` (рекомендуется)
- `docs/CODEOWNERS`

### Когда использовать
- Проекты с несколькими мейнтейнерами
- Код с чёткими областями владения
- Организации с определёнными командами
- Не нужно для сольных проектов

### Структура содержания
- Глобальные шаблоны владения
- Владение, специфичное для директории
- Владение, специфичное для файла
- Назначения команд
- Несколько владельцев на шаблон

## Лучшие практики

1. **FAQ после возникновения вопросов** — Не создавайте пустой FAQ заранее
2. **Примеры бесценны** — Пользователи лучше всего учатся на примерах
3. **ARCHITECTURE для сложных проектов** — Помогает контрибьюторам понять
4. **MIGRATION для несовместимых изменений** — Уважайте время и инвестиции пользователей
5. **Используйте FUNDING.yml GitHub** — Лучшая интеграция, чем пользовательский файл
6. **CODEOWNERS для команд** — Автоматизирует процесс ревью
7. **Ссылайтесь из README** — Сделайте все файлы обнаруживаемыми
8. **Держите обновлёнными** — Устаревшая документация хуже, чем никакой
9. **Документация API должна быть сгенерирована** — Ручная документация API устаревает
10. **VERSION в языко-стандартном месте** — Не создавайте пользовательские файлы
