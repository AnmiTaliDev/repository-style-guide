# Руководства по установке и сборке

Документация по установке, сборке и использованию вашего проекта. Необходима для внедрения пользователями.

## INSTALL

Инструкции по установке для конечных пользователей.

### Требуемое содержание
- Системные требования (ОС, аппаратное обеспечение, зависимости)
- Методы установки (пакетные менеджеры, загрузка бинарников, сборка из исходников)
- Инструкции для конкретных платформ для различных операционных систем
- Шаги проверки после установки
- Руководство по устранению распространённых проблем установки
- Инструкции по удалению
- Процедуры обновления с предыдущих версий

### Когда отделять от README
- Сложный многоступенчатый процесс установки
- Доступны несколько методов установки
- Инструкции для конкретных платформ слишком длинные
- README становится слишком длинным из-за деталей установки

### Варианты названия
- `INSTALL.md`
- `INSTALL`
- `INSTALLATION.md`
- `INSTALLATION`

## BUILD

Инструкции по сборке для разработчиков, компилирующих из исходников.

### Требуемое содержание
- Список зависимостей для сборки с требованиями к версиям
- Требуемые инструменты сборки (компиляторы, системы сборки)
- Пошаговый процесс сборки
- Опции и флаги конфигурации сборки
- Заметки и требования для сборки на конкретных платформах
- Процедуры выполнения тестов после сборки
- Инструкции по отладочной и релизной сборке
- Инструкции по кросс-компиляции (если применимо)
- Руководство по устранению проблем сборки

### Варианты названия
- `BUILD.md`
- `BUILD`
- `BUILDING.md`
- `BUILDING`
- `COMPILE.md`
- `COMPILE`
- `MAKE.md`

### Отличие от INSTALL
- BUILD для разработчиков, собирающих из исходников
- INSTALL для пользователей, устанавливающих предсобранные пакеты

## DEPENDENCIES

Список зависимостей проекта и их назначения.

### Требуемое содержание
- Зависимости времени выполнения с требованиями к версиям
- Зависимости времени сборки
- Опциональные зависимости и их назначения
- Требования к системным библиотекам
- Обоснование для каждой крупной зависимости
- Политика обновления зависимостей
- Известные проблемы совместимости

### Языко-специфичные альтернативы
Многие языки имеют стандартизированные файлы зависимостей, которым следует отдавать предпочтение:
- Python: `requirements.txt`, `Pipfile`, `pyproject.toml`, `setup.py`
- Node.js: `package.json`
- Rust: `Cargo.toml`
- Go: `go.mod`
- Ruby: `Gemfile`
- PHP: `composer.json`

### Когда использовать DEPENDENCIES.md
- Мультиязычные проекты без единой системы зависимостей
- Зависимости системного уровня, не управляемые языковыми инструментами
- Объяснение обоснования выбора зависимостей
- Сложные отношения зависимостей, требующие документации
- Зависимости, которые не могут быть выражены в стандартных файлах

### Варианты названия
- `DEPENDENCIES.md`
- `DEPENDENCIES`
- `REQUIREMENTS.md`
- `REQUIREMENTS`
- `PREREQUISITES.md`
- `PREREQUISITES`

## GUIDE

Подробное руководство по использованию или учебник.

### Требуемое содержание
- Учебник для начинающих
- Распространённые случаи использования с объяснениями
- Документация по продвинутым функциям
- Опции конфигурации
- Лучшие практики
- Руководство по настройке производительности
- Руководства по миграции с похожих инструментов
- Раздел по устранению неполадок

### Когда использовать
- Сложные проекты, требующие детальных пошаговых инструкций
- Альтернатива отдельной директории `docs/` для небольших проектов
- Учебники для быстрого старта
- Не рекомендуется, когда существует полная директория `docs/`

### Варианты названия
- `GUIDE.md`
- `GUIDE`
- `TUTORIAL.md`
- `TUTORIAL`
- `QUICKSTART.md`
- `QUICKSTART`

### Альтернатива: директория docs/
Для больших проектов создайте структурированную директорию документации:
- `docs/getting-started.md`
- `docs/installation.md`
- `docs/configuration.md`
- `docs/api-reference.md`
- `docs/tutorials/`
- `docs/examples/`

## Лучшие практики

1. **INSTALL для пользователей, BUILD для разработчиков** — Чёткое разделение задач
2. **Используйте стандартные файлы зависимостей языка** — Не создавайте пользовательские системы
3. **Ссылайтесь из README** — Сделайте руководства легко обнаруживаемыми
4. **Держите инструкции актуальными** — Регулярно тестируйте их на чистых системах
5. **Включайте шаги проверки** — Помогите пользователям подтвердить успешное завершение
6. **Разделы для конкретных платформ** — Используйте чёткие заголовки и организацию
7. **Предпочтительна установка в одну строку** — Упростите, когда возможно (пакетные менеджеры, установщики)
8. **Совместимость версий** — Документируйте, какие версии работают вместе
9. **Офлайн установка** — Документируйте процедуры для изолированных сред
10. **Регулярно обновляйте** — Инструкции быстро устаревают
